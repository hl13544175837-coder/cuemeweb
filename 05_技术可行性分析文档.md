# 一、项目概述

## 1.1 项目背景

CueMe是夸克的AI智能助手，专注长文本创作。2024年10月新增"小伙伴"功能（角色扮演），旨在将工具型AI转化为情感化、人格化的陪伴型AI。

当前面临三大核心挑战：
- **头部效应过重**：头部角色（秦苒）曝光占比高达50%，导致生态失衡
- **流量分配僵化**：固定推荐位（仅2个角色）限制了流量规模增长
- **互动门槛高**：用户点击后对话转化率低（当前28.2%），流失严重

## 1.2 建设目标

| 维度 | 目标指标 |
|------|----------|
| 流量规模 | 日均曝光量 > 10,000 |
| 点击率 | 整体 CTR ≥ 15% |
| 对话转化 | 点击→对话转化率 ≥ 32% |
| 用户粘性 | 平均对话轮数 ≥ 15 轮；对话留存率（≥3轮）≥ 60% |
| 生态健康 | 头部角色曝光占比 < 30%；UGC角色点击率 ≥ 16% |

## 1.3 建设范围

本项目分四个阶段实施：
- **第一阶段**：流量分发策略重构（动态轮动池 + 赛马机制）
- **第二阶段**：对话转化率提升（互动式首语 + 引导层）
- **第三阶段**：UGC生态闭环（创作者评分体系 + 联动分发）
- **第四阶段**：深度互动与长效记忆（记忆模块 + 剧情机制）
- **附加升级**：多模态沉浸交互（LoRA + 图像生成）

# 三、技术选型分析

## 3.1 推荐算法

### 3.1.1 核心需求

| 需求 | 描述 |
|------|------|
| 动态轮动池 | 支持8-10个角色轮播，基于CTR表现自动调整排序 |
| 赛马机制 | 新角色冷启动保护（500PV）+ CTR熔断降权（<12%） |
| 标签匹配 | 性别标签（男/女/通用）+ 角色标签（热血/高冷等） |

### 3.1.2 推荐算法选型

**方案一：协同过滤 + 热门加权（推荐）**

```
Score(i) = α × HotScore(i) + β × FreshScore(i) + γ × MatchScore(i) + δ × ExploreScore(i)

其中：
- HotScore: 基于CTR的热门得分 (利用)
- FreshScore: 新角色保护得分 (探索)
- MatchScore: 用户-角色标签匹配得分 (精准)
- ExploreScore: 随机探索得分 (长尾)
```

| 优点 | 缺点 |
|------|------|
| 可解释性强，便于人工干预 | 难以捕捉复杂交互特征 |
| 工程实现简单，延迟低 | 冷启动依赖先验数据 |
| 与现有数据链路兼容性好 | 长尾内容分发能力有限 |

**方案二：深度学习排序模型（DIN/DIEN）**

| 优点 | 缺点 |
|------|------|
| 能捕捉用户行为序列特征 | 训练成本高，迭代周期长 |
| 点击率预估精度更高 | 模型可解释性差 |
| 适合大规模特征 | 需要较多训练数据 |

### 3.1.3 推荐方案建议

**采用分层推荐策略**：

| 阶段 | 算法 | 适用场景 |
|------|------|----------|
| 召回层 | 协同过滤 + 标签匹配 | 粗筛候选集（100→20） |
| 精排层 | LightGBM/XGBoost | 特征排序（20→8） |
| 重排层 | 规则引擎（人工干预） | 业务规则（多样性、新角色保护） |

### 3.1.4 技术实现要点

```python
# 推荐服务核心逻辑伪代码
class RecommendationService:
    def get_recommend_list(self, user_id, context):
        # 1. 获取用户画像
        user_profile = self.get_user_profile(user_id)

        # 2. 召回候选集
        candidates = self.recall(user_profile, context, count=100)

        # 3. 精排打分
        scored = self.rank(candidates, user_profile)

        # 4. 重排（加入业务规则）
        final_list = self.rerank(scored, context)

        return final_list[:10]  # 返回Top10

    def rank(self, candidates, user_profile):
        # CTR预估模型
        features = self.extract_features(candidates, user_profile)
        scores = self.ctr_model.predict(features)
        return sorted(zip(candidates, scores), key=lambda x: -x[1])
```

### 3.3.3 关键信息提取流程

```python
class MemoryExtractor:
    def extract_and_store(self, user_id, dialogue_history):
        # 1. 调用LLM提取关键实体
        entities = self.extract_entities(dialogue_history)

        # 2. 分类存储
        for entity in entities:
            if entity.type == "personal_fact":
                self.store_short_term(user_id, entity)  # Redis
            elif entity.type == "long_term_interest":
                self.store_long_term(user_id, entity)   # VectorDB

        # 3. 更新用户画像
        self.update_user_profile(user_id, entities)

    def get_context_for_llm(self, user_id, current_dialogue):
        # 构建发送给LLM的记忆上下文
        short_term = self.get_short_term(user_id)
        long_term = self.get_long_term(user_id, top_k=10)

        return {
            "recent_facts": short_term,
            "user_profile": long_term,
            "reminder_template": "你之前提到过{topic}，现在怎么样了？"
        }
```

### 3.4.1 核心需求

| 功能 | 描述 | 优先级 |

### 3.4.2 LoRA技术方案

**LoRA (Low-Rank Adaptation)** 是一种高效的大模型微调技术：

| 项目 | 规格 |
|------|------|
| 训练成本 | 相比全量微调降低99% |
| 训练时间 | 单角色 30-60分钟（A100 GPU） |
| 模型大小 | 典型权重文件 100-200MB |
| 推理额外延迟 | < 100ms |

# 四、技术难点与风险评估

## 4.1 技术难点分析

### 4.1.1 推荐系统冷启动

| 难点 | 说明 | 应对策略 |
|------|------|----------|
| 新角色无数据 | 新角色进入轮动池时缺乏CTR数据 | 48小时保护期，固定500PV曝光 |
| 探索-利用平衡 | 推高CTR角色 vs 测新角色 | ε-greedy策略，保证10%流量探索 |
| 用户标签缺失 | 新用户无行为数据 | 默认标签池 + 首日行为快速打标 |

**风险等级**：⚠️ 中等
**预计攻克时间**：2-3周

### 4.1.2 AIGC内容质量控制

| 难点 | 说明 | 应对策略 |
|------|------|----------|
| 生成一致性 | 不同角色风格需稳定统一 | 建立角色Prompt库，持续迭代 |
| 违规风险 | 可能生成不当内容 | 人工审核 + 内容过滤双重保障 |
| 响应延迟 | 首语生成需快速响应 | 预生成+缓存，减少在线调用 |

**风险等级**：⚠️ 中等
**预计攻克时间**：1-2周

### 4.1.3 长记忆模块构建

| 难点 | 说明 | 应对策略 |
|------|------|----------|
| 信息提取准确性 | LLM提取关键信息的准确率 |few-shot prompt优化 + 人工抽样验证 |
| 记忆冲突处理 | 用户前后说法矛盾 | 以最近一次为准，标记置信度 |
| 存储成本 | 大量用户记忆的存储开销 | 冷热分层，重要信息优先 |

**风险等级**：🔴 高
**预计攻克时间**：3-4周

### 4.1.4 LoRA训练与推理

| 难点 | 说明 | 应对策略 |
|------|------|----------|
| 训练资源 | LoRA训练需要GPU资源 | 初期使用夸克云GPU资源池 |
| 形象一致性 | 不同场景下角色脸不变 | LoRA权重0.6-0.8，避免过拟合 |
| 生成速度 | 实时生图延迟影响体验 | 异步生成 + 图片预加载 |

**风险等级**：🔴 高
**预计攻克时间**：4-6周

### 4.1.5 系统性能与延迟

| 场景 | 目标延迟 | 挑战 |
|------|----------|------|
| 列表页推荐 | < 50ms | 需要缓存+异步更新 |
| 首语生成 | < 500ms | LLM调用延迟 |
| 对话响应 | < 1000ms | 记忆检索+LLM生成 |
| 图像生成 | < 5s | SDXL推理耗时 |

**风险等级**：⚠️ 中等
**预计攻克时间**：持续优化

# 八、结论与建议

## 8.1 可行性结论

| 维度 | 评估 | 说明 |
|------|------|------|
| **技术可行性** | ✅ 可行 | 核心算法和架构方案成熟，无技术阻塞 |
| **资源可行性** | ⚠️ 需协调 | GPU资源需提前申请，LoRA阶段有资源风险 |
| **时间可行性** | ⚠️ 紧张 | 16周周期较长，建议优先级排序 |
| **成本可行性** | ✅ 可控 | 相比产品潜在收益，330万投入合理 |

## 8.2 优先级建议

| 优先级 | 阶段 | 理由 |
|--------|------|------|
| **P0** | 第一阶段 | 解决核心流量问题，见效快 |
| **P0** | 第二阶段 | 直接提升对话转化，业务价值明确 |

| **P2** | 第四阶段 | 技术复杂度高，建议充分预研 |
| **P3** | 多模态升级 | 投入大、风险高，建议后期再上 |
